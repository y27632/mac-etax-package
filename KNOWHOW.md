# .xtx 生成ノウハウ集

e-Tax WEB版向け法人税申告XTXファイル生成で得られた知見をまとめたものです。

---

## 1. フォームバージョンは RHO スキーマで必ず確認する

`ETaxDocument("RHO0022", "25.0.2", ...)` のように手続きコードのバージョンを指定すると、
e-Tax側は対応する **RHO0022-250.xsd** でバリデーションを行う。

このXSDに `<xsd:include schemaLocation="HOA116-005.xsd"/>` のように**各フォームの必須バージョンが明記**されている。
`specs/xsd/RHO0022-250.xsd` を開いて `schemaLocation` を検索し、使用バージョンと一致しているか必ず確認すること。

```
# 第4期（RHO0022 v25.0.2）の対応バージョン例
HOA112 → 6.0
HOA116 → 5.0   ← 4.0 を使うと「XML構造エラー」になる
HOA201 → 5.0
HOA420 → 24.0
HOA511 → 15.0
HOA522 → 9.0
HOB016 → 6.0
HOK010 → 7.0
```

## 2. XSD の型定義（複合型 vs 単純型）を必ず確認する

タグが存在していても、**複合型コンテナに単純型の値を直接渡すとスキーマエラー**になる。
新しいタグを追加する前に `specs/xsd/` のXSDで型定義を確認すること。

```python
# ❌ 間違い（VAB00040 は複合型コンテナ）
"VAB00040": 1

# ✅ 正しい（実値は子要素 VAB00060 に入れる）
"VAB00040": {"VAB00060": 1}
```

詳細な誤り事例は `memory/feedback_etax_xsd_structure.md` を参照。

## 3. 法人事業概況説明書（HOK010）の月次テーブル構造

`IAP20000` は月次行のリスト。各行の構造：

```python
{
    "IAP20100": 2,                    # 月（2〜1で12ヶ月）
    "IAP20200": {"IAP20300": 426714}, # 当期売上金額（IAP20300 = 金額1）
    "IAP20800": 1                     # 従事員数
}
```

- `IAP20300` = 当期売上（金額1）
- `IAP20400` = 売上金額2（月次行への追加は不要。前期合計は IAP40000 へ）
- `IAP20800` = **従事員数**（役員のみでも `1` を入力すること）

**前期の実績（合計）の入力場所：**

```python
"IAP30000": {"IAP30100": {"IAP30200": 426714}},  # 当期計
"IAP40000": {"IAP40100": {"IAP40200": 1083130}}  # 前期計 ← ここ
```

`IAP30300` を `IAP30100` の中や `IAP30000` の兄弟に置いても正しく表示されない。

## 4. 別表五(一) ICB00400 繰越損益金の増減構造

```python
"ICB00400": {
    "ICB00410": 900883,   # 期首
    "ICB00420": {
        "ICB00430": 900883,   # 減（期首と同額で振り替え）
        "ICB00440": 1203605   # 増 本書き（累積純利益）
    },
    "ICB00450": 1203605   # 期末
}
```

`ICB00420`（増減）は複合型コンテナ。`ICB00430` = 減、`ICB00440` = 増本書き。
当期純損失の場合は増欄（`ICB00440`）に**マイナス値**を入れる。

## 5. 別表五(二) IEC の行数制限

`IEC00010`（道府県民税 前期以前行）は **maxOccurs=2**。
複数期分をまとめる場合は事業年度を「最初の期〜最後の期」に合算し、1〜2行に収める。

## 6. 別表二（HOA201）の構造的注意点

### バージョン

RHO0022 v24.0.2 では HOA201 のバージョンは **5.0**（テンプレート記載の 4.0 は誤記）。

```python
doc.add_form("HOA201", "5.0", {...})  # ✅
doc.add_form("HOA201", "4.0", {...})  # ❌ スキーマエラー
```

### VAE00000 は単一要素（リストではない）

社員が1人だけでも `VAE00000` はリストにしない。2人目以降は `VAE00170`（maxOccurs=12）に入れる。

```python
# ❌ 間違い（リストとして渡すとエラー）
"VAE00000": [{"VAE00010": {...}}, {"VAE00010": {...}}]

# ✅ 正しい構造
"VAE00000": {
    "VAE00010": {                       # 1人目（本人）
        "VAE00020": {"VAE00030": 1, "VAE00040": 1},   # 順位
        "VAE00050": {"VAE00060": "<住所>", "VAE00070": "<氏名>"},
        "VAE00080": {
            "VAE00120": {               # その他の株主等
                "VAE00130": <出資額>,
                "VAE00140": {"VAE00160": <議決権数>},
            }
        },
    },
    "VAE00170": [                       # 2人目以降（リスト、最大12エントリ）
        {
            "VAE00180": {"VAE00190": 2, "VAE00200": 2},
            "VAE00210": {"VAE00220": "<住所>", "VAE00230": "<氏名>"},
            "VAE00250": {
                "VAE00290": {           # その他の株主等
                    "VAE00300": <出資額>,
                    "VAE00310": {"VAE00330": <議決権数>},
                }
            },
        },
        # 追加社員は同じ構造で繰り返す
    ],
}
```

### 判定割合タグは simpleContent → 直接値でOK

VAB00030 / VAB00110 / VAB00120 / VAC00020 / VAC00060 / VAC00070 はすべて `simpleContent` 型。
整数値を直接渡す（dict にしない）。

```python
"VAB00030": 100,   # ✅ 整数で直接渡す
"VAB00030": {"value": 100},  # ❌ エラー
```

## 7. 別表十五（HOE200）の要素順序

### EGE00000 は EGD の直後・EGF の前に配置する

XSD のsequence定義: EGA → EGB → EGH → EGC → EGD → **EGE** → EGF → EGG

EGE00000（損金不算入額）が0の場合は省略できるが、配置する場合は EGD00000 の直後でなければ「XML構造チェックエラー」になる。

```python
# ✅ 正しい順序（EGEはEGFより前）
{
    "EGD00000": {"EGD00005": {"kubun_CD": "1"}, "EGD00020": 8_000_000},
    "EGE00000": 2_000_000,   # ← EGD直後・EGF前
    "EGF00000": {...},
}

# ❌ 間違い（EGEをEGFより後に配置）
{
    "EGD00000": {...},
    "EGF00000": {...},
    "EGE00000": 2_000_000,   # ← XSD順序違反 → スキーマエラー
}
```

Python dict は挿入順を保持するため、辞書を組み立てる際に EGE を EGF より先に追加すること。

## 8. 法人事業概況説明書（HOK010）PDFの単位は千円

HOK010の帳票PDF（損益・貸借各欄）は**単位が千円**。XTXには円単位で値を入れるが、PDF帳票には ÷1000（切り捨て）の値が表示される。

```
# 例: XTX値 4,900,568 → PDF帳票には "4,900"（千円）
"IAI01100": 4_900_568   # → PDF: 4,900
"IAI01610": 2_400_000   # → PDF: 2,400
"IAI01630": 30_000      # → PDF: 30
```

verify_xtx.py での照合は自動でこの換算を行う（HOK010フィールドのみ÷1000）。

## 9. 第2期以降の前期データ引き継ぎ（繰越残高の設定方法）

2期目以降の申告では、前期末の残高を当期の期首値として引き継ぐ必要がある。

### 別表五(一) の前期残高引き継ぎ

| 前期末（n-1期の期末値） | 当期（n期）の期首値タグ |
|---|---|
| ICB00450 繰越損益金期末 | ICB00410 繰越損益金期首 |
| ICB00610 未納法人税及び地方法人税期末（マイナス値） | ICB00540 同期首 |
| ICB00700 未納道府県民税期末（マイナス値） | ICB00630 同期首 |

```python
# 第2期（前期末からの引き継ぎ例）
icb00410 = 前期_ICB00450    # 前期末の繰越損益金（例: 1,203,605）
icb00540 = 前期_ICB00610    # 前期末の未納法人税（例: -158,200）
icb00630 = 前期_ICB00700    # 前期末の未納道府県民税（例: -50,000）
```

### 別表五(二) の前期繰越行（IEB00010 / IEC00010）

前期以前に未払のまま残っている租税は `IEB00010`（法人税用）/ `IEC00010`（道府県民税用）に記載する。

```python
# IEB00010: 前期以前の未払法人税行（maxOccurs=2）
"IEB00010": {
    "IEB00020": "5,06,02,01",   # 前期の事業年度（自）
    "IEB00030": "5,07,01,31",   # 前期の事業年度（至）
    "IEB00050": 158_200,         # 期首現在未納税額
    "IEB00160": 158_200,         # 期末現在未納税額（未納付のまま）
}
```

> **注意**: 前期に納付済みの場合は `IEB00160`（期末未納額）を 0 にする。
> 複数期分（3期以上前も未納）がある場合は最大2行に合算して記載（maxOccurs=2）。

### データ引き継ぎの実例（gen_step_c_integration.py）

`tests/gen_step_c_integration.py` の `case1-full-integration` が第2期・前期残高あり・全別表統合の検証済みサンプル（2026-04-16検証済み・PDF照合97件全一致）。

## 10. 複数別表を1本の XTX に含める場合はフォームの追加順序が必須

`doc.add_form()` を呼ぶ順序は **XSD の sequence 定義通り**でなければならない。

`specs/xsd/RHO0022-240.xsd`（または使用バージョンの XSD）の `<xsd:group ref="...">` の出現順序が厳密に要求される。

```
# RHO0022 v24.0.2 の正しい順序
1. HOA112（別表一）         ← minOccurs=1 必須
2. HOA116（別表一次葉）
3. HOA201（別表二）
4. HOA420（別表四）
5. HOA511（別表五(一)）
6. HOA522（別表五(二)）
7. HOE200（別表十五）       ← HOB016（別表六(一)）ではない
8. HOK010（法人事業概況説明書）
```

**よくある間違い**:
- HOA201（別表二）を最初に追加する → 「XML構造チェックエラー」
- 申告マスター.md の旧版では順序が誤っていた（別表二→別表四→別表十五の順）← 2026-04-16修正済み
- 別表十五を HOB016（別表六(一)の間違い）と誤記 → 正しくは HOE200

## 11. XSD ファイルの取得方法

- ダウンロード元: https://www.e-tax.nta.go.jp/shiyo/shiyo3.htm の `e-tax19.CAB`
- macOS では `cabextract` で展開できるが、日本語ファイル名は文字化けする場合がある
- 内容一覧: `cabextract -l e-tax19.CAB` で確認し、`cabextract -p e-tax19.CAB -F "*/ファイル名.xsd" > 出力.xsd` で個別抽出
- 必要なXSDを個別に `specs/xsd/` へ配置して参照する
- `specs/README.md` にバージョン対応表を記載

## 12. 別表間のデータ連携（転記ルール）

複数の別表を組み合わせる際に、ある別表の計算結果を別の別表に転記する必要がある。

| 転記元 | タグ | 転記先 | タグ | 説明 |
|--------|------|--------|------|------|
| 別表七(一) | MCB00240 | 別表四 | ART00010 | 欠損金の損金算入額（当期控除額合計）|
| 別表六(一) | FZC00330 / FZG集計 | 別表一 | BGB00180 | 源泉徴収税額の控除額合計 |
| 別表四 | ARU00170 | 別表一 | BGB00010 | 所得の金額（または欠損金額） |
| 別表一次葉 | QHC00010 | 別表一 | BGB00020 | 法人税額の合計 |
| 別表一次葉 | QHC00050 | 別表一 | BGC00060 | 地方法人税額 |

### 別表七(一) → 別表四 の連携

```python
# 別表七(一) MCB00240 = 当期控除額合計（例: 500,000）
# → 別表四 ART00010 に転記（欠損金等の当期控除額・留保）

# 別表四
"ART00000": {
    "ART00010": {
        "ART00020": 500_000,  # 留保（MCB00240 と同額）
    }
}
```

### 別表六(一) → 別表一 の連携

```python
# FZC00330（預貯金利子の所得税等計）+ FZG00010 の所得税等合計 → BGB00180
# 例: FZC00330=76,575 → BGB00180=76,575
"BGB00220": {
    "BGB00230": 76_575,   # 所得税額
    "BGB00250": 76_575,   # 計
    "BGB00260": 76_575,   # 控除した金額
},
"BGB00180": 76_575,       # 別表一の源泉徴収税額
```

## 13. 別表六(一)（HOB016）の構造的注意点

### 受取利子の種類ごとの使い分け

| 種類 | 使うタグ | 備考 |
|------|---------|------|
| 預貯金・公社債の利子 | `FZC00010` | 単一要素。所得税等は `FZC00310`/`FZC00320` |
| その他（配当等） | `FZG00010` | **リスト形式**（maxOccurs=5）|

### FZC（預貯金利子）の集計構造

```python
"FZC00000": {

        "FZC00330": 76_575,  # ← BGB00180 に転記する値
    },
}
```

### FZG（その他）はリスト

```python
"FZG00000": {
```

## 14. 別表七(一)（HOB710）の構造的注意点

### 欠損金行の最初の1件と2件目以降で構造が異なる

```
MCB00010 = 最初の欠損金行（単一要素）
MCB00110 = 2件目以降の欠損金行（リスト形式、maxOccurs=9）
```

```python
# 欠損金が1期分のみの場合（MCB00010 のみ使用）
"MCB00010": {
    "MCB00020": "5,05,02,01",  # 欠損事業年度（自）
    "MCB00030": "5,06,01,31",  # 欠損事業年度（至）
    "MCB00060": 300_000,       # 繰越控除前欠損金額
    "MCB00220": 300_000,       # 当期控除額
    "MCB00260": 0,             # 翌期繰越額
}

# 欠損金が2期分以上の場合（MCB00110 を追加）
"MCB00110": [
    {
        "MCB00120": "5,04,02,01",
        "MCB00130": "5,05,01,31",
        "MCB00160": 400_000,
        "MCB00370": 200_000,   # 当期控除額（2件目以降は MCB00370）
        "MCB00380": 200_000,   # 翌期繰越額（2件目以降は MCB00380）
    }
]
```

> **注意**: `MCB00220`（当期控除額）は MCB00010 用。MCB00110 の行では `MCB00370`。

### 当期控除額の合計を別表四に転記

```python
# MCB00240 = 当期控除額合計 → 別表四 ART00010
"MCB00230": {
    "MCB00240": 500_000,  # 合計（全行の当期控除額の合計）
}
```

## 15. 中間申告（予定申告）の注意点

### 手続きコードが確定申告とは異なる

| 申告種別 | 手続きコード | 主な帳票 |
|---------|-------------|--------|
| 確定申告（青色） | RHO0022 | HOA112, HOA116, HOA420 等 |
| 予定申告（中間申告） | **RHO0061** | **HOE701（別表十九）のみ** |
| 確定申告（白色） | RHO0021 | 別途帳票 |

```python
# 予定申告は RHO0061 で ETaxDocument を作成
doc = ETaxDocument("RHO0061", "25.0.0", it_info)
doc.add_form("HOE701", "6.0", {...})  # 1帳票のみ
```

### 月数換算（RZD00055 / RZE00055）

`RZD00055` は前事業年度の**月数**（通常12）。simpleContent 型なので直接整数を渡す。

```python
"RZD00055": 12,   # ✅ 整数で直接渡す
```

納付すべき税額の計算（百円未満切り捨て）:

```python
rzd00070 = (rzd00050 * 6 // rzd00055 // 100) * 100
rze00070 = (rze00050 * 6 // rze00055 // 100) * 100
```

## 16. AIが犯しやすいミスパターン一覧

| # | ミスパターン | 症状 | 正しい対応 |
|---|-------------|------|-----------|
| 1 | フォームバージョンを手動で指定する | XML構造エラー | RHO0022-xxx.xsd の `<xsd:include>` で必ず確認（§1参照）|
| 2 | 複合型コンテナに値を直接渡す | XML構造エラー | XSDで型定義を確認してから辞書構造を組み立てる（§2参照）|

| 4 | simpleContent 型を dict にする | XML構造エラー | `{"value": 100}` ではなく `100` を直接渡す（§6参照）|
| 5 | HOA201 の VAE00000 をリストにする | XML構造エラー | 1人目は単一要素、2人目以降は VAE00170 リスト（§6参照）|

| 9 | HOK010 の金額をそのままPDFと照合する | 照合失敗 | PDFは千円単位表示（XTX値÷1000）（§8参照）|
| 10 | 別表十五 EGE00000 を EGF00000 の後に置く | XML構造エラー | XSD sequence 通り EGD → EGE → EGF の順（§7参照）|

## 17. デバッグ手順（e-Taxエラーメッセージ別の対処法）

### エラー1: 「XML構造チェックエラーです。読み込んだ申告・申請データのXML構造に誤りがあります」

帳票数が 0 件でインポートされるのが特徴。原因は3つ。

**原因A: フォームの追加順序が XSD と違う**
```
対処: specs/xsd/RHO0022-xxx.xsd を開いて <xsd:group ref> の出現順序を確認
     doc.add_form() の呼び出し順を XSD と一致させる（§10参照）
診断: 正常にインポートできる既存 XTX と diff して FORM_SEC の rdf:li 順序を比較
```

**原因B: タグの構造（複合型/単純型）が XSD と違う**
```
対処: 該当帳票の XSD（例: HOA511-015.xsd）を開いてタグの型を確認
     複合型コンテナには子要素を辞書で、単純型には値を直接渡す（§2参照）
```

**原因C: フォームバージョン（VR）が間違っている**
```
対処: RHO0022-xxx.xsd の <xsd:include schemaLocation="..."> でバージョンを確認
     e-Tax は VR が XSD と一致しないと構造エラーを返す（§1参照）
```

---

### エラー2: 「ファイルの読み込みに失敗しました」（インポートダイアログで即時エラー）

**原因: XML自体が不正（エンコーディング、タグ未閉鎖など）**
```
対処: python3 -c "import xml.etree.ElementTree as ET; ET.parse('申告.xtx')" で構文チェック
     etax_builder.py の xml_escape が正しく適用されているか確認
     特殊文字（& < > "）が & &lt; &gt; &quot; にエスケープされているか確認
```

---

### エラー3: 帳票はインポートされるが PDF の値が空欄または不一致

**原因A: IDREF の参照先 ID が IT ヘッダに存在しない**
```
症状: 法人名や住所が PDF に表示されない
対処: "REF:NOZEISHA_NM" → IT ヘッダの <NOZEISHA_NM ID="NOZEISHA_NM"> と対応するか確認
```

**原因B: 金額タグに 0 を入れている（省略すべき箇所）**
```
症状: 0 の欄が意図せず埋まる（または計算合計がずれる）
対処: 0 値は None または省略で OK。verify_xtx.py も 0 はスキップして照合
```

**原因C: 日付形式が正しくない**
```
症状: 事業年度が空欄になる
対処: "5,07,02,01" 形式（元号番号,年,月,日）を使う
     令和=5, 平成=4, 昭和=3。年は2桁ゼロ埋め（例: 令和7年→"07"）
```

### デバッグの基本手順

```
1. python3 -c "import xml.etree.ElementTree as ET; ET.parse('申告.xtx')" で XML 構文確認
2. e-Tax WEB版にインポートして帳票数を確認（0件 = XML構造エラー）
3. 帳票数が正常でもPDF値が不一致なら verify_xtx.py --check で差分確認
4. 既知の正常 XTX（tests/step-b/*/data.xtx）と XML 構造を diff で比較
5. specs/xsd/ の XSD でタグの型と順序を直接確認
```
